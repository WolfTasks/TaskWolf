# i18n Rollout — Session 3 (`comments` namespace) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Localize the S3 `comments` slice — the issue Comments/Activity tabs, the comment thread, and the activity feed — into a new `comments` namespace (`en`/`de`), and **roll out relative-time formatting via `format.ts`** (the first real use of `formatRelativeTime`), then flip S3 to ✅ in the master-spec coverage matrix.

**Architecture:** Thin execution plan against the locked pattern in `docs/superpowers/specs/2026-07-13-i18n-full-rollout-design.md` (Anhang — Migrations-Checkliste). Add the namespace first, then localize the three components. Two of them (`CommentThread`, `ActivityFeed`) each hand-roll a local `formatTime` that concatenates `toLocaleDateString()` + `toLocaleTimeString()`; both are replaced by `formatRelativeTime` from `@/i18n/format`, satisfying DoD Abschnitt 5.3 and the master-spec Abschnitt 8 "relative-Zeit-Rollout". The activity feed's `describeActivity` message map moves to interpolated i18next keys.

**Tech Stack:** react-i18next (`useTranslation`), `frontend/src/i18n/format.ts` (`formatRelativeTime`), the Session-0 scanner/parity scripts, Node 20, Vite/tsc build.

## Global Constraints

- **Namespace = `comments`.** In components use `const { t } = useTranslation('comments')`; reference shared terms with the `common:` prefix (`t('common:save')`). Unprefixed keys resolve against `comments`.
- **en/de stay key-identical** (parity check is authoritative).
- **Relative-time rollout is a core deliverable of this slice.** Both `CommentThread` and `ActivityFeed` currently define a private `formatTime(iso)` that renders an absolute date+time (`d.toLocaleDateString() + ' ' + d.toLocaleTimeString(...)`). Delete both and render `formatRelativeTime(createdAt)` from `@/i18n/format`. This is an **intentional UX change** ("2 hours ago" / "vor 2 Stunden") — `Intl.RelativeTimeFormat` handles the DE/EN wording itself, so no translation keys are needed for the timestamps. **No raw `toLocaleString`/`toLocaleDateString` may remain** in these files.
- **Scanner-blind but DoD-required strings** (localize even though the scanner does NOT flag them): the `CommentsActivityTabs` tab labels `'Comments'/'Activity'` (they live in a `cond ? 'A' : 'B'` JSX expression, not JSX text) and the `CommentThread` `confirm('Delete this comment?')` and its `? :` button labels (`'Posting...'`/`'Post Comment'`, `'Loading...'`/`'Load older comments'`), and `ActivityFeed`'s `describeActivity` return strings + its `? :` labels (`'Loading...'`/`'Load more'`).
- **No string-concatenation of translated fragments.** Interpolate via variables (`t('activity.STATUS_CHANGED', { from, to })`). The `(edited)` marker renders as `` ` ${t('thread.edited')}` `` (literal leading space for layout, not a joined fragment). The `—` em-dash fallback in activity values and the `•`/dot glyphs stay literal.
- **Reuse `common:` for shared terms:** `common:save`, `common:cancel`, `common:loading` (already exist). Do not duplicate them into `comments`.
- **Remove each localized file from `frontend/scripts/i18n-allowlist.json`** in the same task that localizes it. The allowlist only shrinks. **Net for S3: 2 files removed (54 → 52).** `CommentThread.tsx` and `ActivityFeed.tsx` are in the allowlist; `CommentsActivityTabs.tsx` is **not** (it is scanner-blind), so it needs no allowlist edit.
- **Done-per-slice (master spec Abschnitt 5):** `npm run test:i18n && npm run lint:i18n && npm run build` all green; manual DE/EN browser check; matrix row flipped to ✅.

- [ ] **Step 1: Replace the entire file**

The module-level `formatTime` and `describeActivity` are removed; `describeActivity` becomes a closure (`describe`) inside the component so it can use the `t` hook, and the message strings become interpolated `comments:activity.*` keys. Replace the entire contents of `frontend/src/components/comments/ActivityFeed.tsx` with:

```tsx
import type { ActivityItem } from '@/types'
import { useTranslation } from 'react-i18next'
import { useActivity } from '@/hooks/useComments'
import { formatRelativeTime } from '@/i18n/format'

interface Props {
  projectKey: string
  issueKey: string
}

export function ActivityFeed({ projectKey, issueKey }: Props) {
  const { t } = useTranslation('comments')
  const { data, isLoading, fetchNextPage, hasNextPage, isFetchingNextPage } = useActivity(projectKey, issueKey)

  const describe = (item: ActivityItem): string => {
    switch (item.type) {
      case 'COMMENT': return t('activity.COMMENT')
      case 'STATUS_CHANGED': return t('activity.STATUS_CHANGED', { from: item.oldValue, to: item.newValue })
      case 'ASSIGNED': return t('activity.ASSIGNED', { to: item.newValue })
      case 'UNASSIGNED': return t('activity.UNASSIGNED', { from: item.oldValue })
      case 'PRIORITY_CHANGED': return t('activity.PRIORITY_CHANGED', { from: item.oldValue, to: item.newValue })
      case 'TITLE_CHANGED': return t('activity.TITLE_CHANGED', { to: item.newValue })
      case 'DESCRIPTION_CHANGED': return t('activity.DESCRIPTION_CHANGED')
      case 'STORY_POINTS_CHANGED': return t('activity.STORY_POINTS_CHANGED', { from: item.oldValue ?? '—', to: item.newValue })
      case 'DUE_DATE_CHANGED': return t('activity.DUE_DATE_CHANGED', { from: item.oldValue ?? '—', to: item.newValue ?? '—' })
      case 'SPRINT_CHANGED': return t('activity.SPRINT_CHANGED', { to: item.newValue ?? t('activity.backlog') })
      case 'ATTACHMENT_ADDED': return t('activity.ATTACHMENT_ADDED', { to: item.newValue })
      case 'ATTACHMENT_REMOVED': return t('activity.ATTACHMENT_REMOVED', { from: item.oldValue })
      default: return (item.type as string).toLowerCase().replace(/_/g, ' ')
    }
  }

  // Pages arrive newest-first; keep that order for the activity log.
  const items: ActivityItem[] = (data?.pages ?? []).flatMap(p => p.content)

  if (isLoading) return <div className="text-gray-500 text-sm">{t('feed.loading')}</div>

  if (items.length === 0) {
    return <p className="text-gray-600 text-sm italic">{t('feed.empty')}</p>
  }

  return (
    <div className="max-h-[26rem] overflow-y-auto space-y-2 pr-1">
      {items.map((item: ActivityItem) => (
        <div key={item.id} className="flex gap-2 items-start text-sm">
          <div className="w-1.5 h-1.5 rounded-full bg-gray-600 mt-1.5 flex-shrink-0" />
          <div>
            <span className="text-gray-300">{describe(item)}</span>
            <span className="text-gray-600 ml-2 text-xs">
              {formatRelativeTime(item.createdAt)}
            </span>
          </div>
        </div>
      ))}

      {hasNextPage && (
        <button
          onClick={() => fetchNextPage()}
          disabled={isFetchingNextPage}
          className="text-xs text-indigo-400 hover:text-indigo-300 disabled:opacity-50"
        >
          {isFetchingNextPage ? t('common:loading') : t('feed.loadMore')}
        </button>
      )}
    </div>
  )
}
```

> Note: the `default:` branch keeps the humanized-type fallback for unknown activity types (a defensive path, not a normally user-visible string) — it is a computed transform of `item.type`, not a hardcoded literal, so the scanner does not flag it and no key is needed.
